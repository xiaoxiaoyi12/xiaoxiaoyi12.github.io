## AI 辅助记录

### 对独立思考的 Review

#### 1. 功能盘点 — 基本到位，有遗漏

你列出了 7 个功能，覆盖了核心场景。但有几个隐含功能没有盘到：

- **用户系统**（注册/登录/个人设置）— 这是后端化的前提，不是"功能"但必须有
- **AI 对话历史** — AI 助手的对话记录要不要持久化？换设备能不能看到之前的对话？
- **网页剪藏的原始内容** — 剪藏的网页 HTML/文本存在哪里？这部分数据可能很大

> 建议：功能盘点时可以用"用户操作流"来检查遗漏 —— 从注册开始，一步步走完所有操作，每个操作涉及什么数据。

#### 2. 数据分析 — 方向对，但分析不够深

你正确识别了需要同步的 4 类数据，也知道它们现在存在 IndexedDB 中。但计划要求的是区分**哪些数据换电脑会心疼**，你没有做优先级区分：

| 数据 | 丢了心疼吗？ | 同步优先级 |
|------|------------|-----------|
| 书籍文件本身（EPUB/PDF） | 很心疼（可能是付费的、找了很久的） | 高 |
| 笔记和高亮 | 最心疼（自己的思考，无法重建） | 最高 |
| 阅读进度 | 有点烦（但重新翻到那一页就行） | 中 |
| 放大/缩小偏好 | 无所谓 | 低，可以只存本地 |
| AI 对话历史 | 看情况 | 低，先不做 |

> 这种优先级思维在后面做同步策略时会用到 —— 高优先级的数据要保证不丢（立即保存），低优先级的可以延迟同步。

#### 3. 架构草图 — 这是今天最需要改进的部分

**问题 1：跳过了架构直接设计接口**

计划要求画的是"前端 ↔ API 层 ↔ 数据库"的系统架构图，你写的是具体的接口列表。这两个是不同层次的东西：

```
架构图（你应该先画这个）：
┌──────────┐    HTTP     ┌──────────┐    SQL      ┌──────────┐
│  React   │ ◄────────► │  NestJS  │ ◄────────► │ PostgreSQL│
│  前端     │            │  后端 API │   Prisma   │  数据库    │
└──────────┘            └──────────┘            └──────────┘
                              │
                         文件存储（本地/S3）

接口列表（架构确定后才设计这个）：
POST /books, GET /books/:id, ...
```

**问题 2：保存机制的思考很好，但方案有坑**

你提出了两个方案：手动保存 vs 定时保存。这个思考方向是对的。但接口设计有问题：

> "调用 save-book 接口，传递 ID 及当前 book **所有数据**"

这是一个**全量覆盖**的方案，问题很多：
- 一本书有 200 条高亮，每次保存都传 200 条，90% 没改过的也传了 —— 浪费带宽
- 如果用户在两台设备上同时操作，全量覆盖会丢数据

更好的方案是**增量操作**：
- 新增一条笔记 → `POST /books/:id/notes`（只传这一条）
- 修改一条高亮 → `PATCH /highlights/:id`（只传改了的字段）
- 删除 → `DELETE /notes/:id`
- 这样每次操作只传变化的部分，冲突也更容易处理

**问题 3：架构图缺少认证层**

用户请求到后端之间，需要一层认证：
- 哪些接口需要登录才能访问？（几乎所有）
- 哪些不需要？（注册、登录、分享页面）
- 认证信息（JWT token）怎么传递？

#### 4. 技术选型 — 结论对，理由太薄

你最终选了 NestJS + PostgreSQL + Prisma，这个选择完全正确。但理由需要更扎实：

- Express 的问题不只是"需要找中间件"，更关键的是**没有统一的代码组织方式**，项目一大就乱
- MongoDB 你说"关系复杂处理起来麻烦"——更准确的说法是：MongoDB 没有原生 JOIN，多对多关系（比如书和标签）需要手动维护引用，而且多文档事务性能差
- Prisma vs TypeORM 你完全没有对比（计划里要求了）—— Prisma 的核心优势是**从 Schema 自动生成完整的 TypeScript 类型**，查询结果不需要手动标注类型
- PostgreSQL 的优势你没提到 `jsonb`（适合存高亮定位信息）和内置全文搜索

> 技术选型的关键不是记住"A 比 B 好"，而是理解**在什么场景下 A 比 B 好**。同样的技术在不同项目中可能得出相反的结论。
