# V3.2 设计规格：Admin 后台改进

## 目标

提升博客管理后台的可用性、数据洞察能力和安全性。聚焦 5 个核心改进：列表排序分页、媒体库、增强仪表盘、密码保护、frontmatter 解析健壮化。

## 技术决策记录

| 决策 | 选项 | 选定 | 理由 |
|------|------|------|------|
| 列表分页 | 前端分页 / API 分页 | **前端分页** | 文章 <200 篇，数据已一次加载，前端排序分页零额外请求 |
| 图表渲染 | Chart.js / ECharts / 纯 CSS | **纯 CSS** | 零依赖，统计图形简单（条形图、热力图），不需要交互式图表库 |
| 密码保护 | 前端密码门 / HTTP Basic Auth / 无 | **前端密码门** | 静态站无后端，前端方案足以防止无意访问 |
| 密码哈希 | Web Crypto SHA-256 / bcrypt.js | **Web Crypto SHA-256** | 浏览器原生 API，零依赖 |
| 前端 YAML 解析 | 自写解析器 / js-yaml / yaml | **js-yaml** | 最成熟的浏览器端 YAML 库，~4KB gzip，兼容性好 |
| Markdown 预览 | ~~添加预览切换~~ | **不做** | Tiptap 本身是所见即所得，预览多余 |

## 模块 A：文章列表排序 + 分页

### 现状

所有文章一次加载，按日期降序固定排列，无排序控件，无分页。

### 设计

在 admin 列表页表头添加可点击排序列：

- **日期**（默认）— 点击切换升/降序
- **标题** — 按中文拼音 / 英文字母排序（使用 `localeCompare('zh-CN')`）
- 当前激活排序列显示箭头指示器（↑ / ↓）

前端分页：

- 每页 20 条
- 底部分页器：上一页 / 下一页按钮 + 当前页码 / 总页数
- 搜索、类型筛选改变时重置到第 1 页
- 数据一次加载后在内存中排序分页

### 受影响的文件

- 修改：`src/app/admin/page.tsx`（添加排序状态 + 分页逻辑 + 分页器 UI）

## 模块 B：媒体库

### 现状

图片通过编辑器上传到 `public/images/{type}/{YYYY-MM}/`，没有集中管理界面。上传后只能通过 GitHub 仓库页面查看。

### 设计

新增 `/admin/media/` 页面，侧边栏添加"媒体库"导航项。

**数据获取：** 复用 `getRepoTreeRecursive()`，筛选 `public/images/` 下的 blob 文件。从 tree entry 中提取文件名、路径、sha。图片缩略图 URL 通过 GitHub raw content URL 拼接：`https://raw.githubusercontent.com/{owner}/{repo}/{branch}/{path}`。

**页面布局：** 网格卡片（响应式 grid），每张卡片包含：
- 缩略图（固定高度 160px，object-fit: cover）
- 文件名（截断显示）
- 文件路径

**操作（点击卡片展开操作面板）：**
1. **复制 Markdown** — 生成 `![filename](/images/...)` 并复制到剪贴板
2. **复制 URL** — 复制 `/images/...` 路径
3. **删除** — 确认后调用 `deleteFile` 从仓库删除

**不做的功能：**
- 引用检测（需要解析所有文章内容匹配图片路径，复杂度高）
- 图片上传（已有编辑器内拖拽上传）
- 文件夹筛选/搜索（初始版本文件少，不需要）

### 受影响的文件

- 新建：`src/app/admin/media/page.tsx`
- 修改：`src/components/admin/AdminSidebar.tsx`（添加"媒体库"导航项）

## 模块 C：增强仪表盘

### 现状

文章列表页顶部有简单统计：总数 + 各类型数量 + 7 天活动柱状图（内联在 page.tsx 中）。

### 设计

将仪表盘拆分为独立组件 `AdminDashboard`，展示四个统计区块：

#### 1. 总览卡片（数字统计）

```
┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐
│  总文章  │ │ 本月新增  │ │  总字数   │ │ 连续写作  │
│    43    │ │    8     │ │  12.3万   │ │   7 天   │
└──────────┘ └──────────┘ └──────────┘ └──────────┘
```

- 总文章数：直接计数
- 本月新增：筛选当月日期的文章数
- 总字数：从已加载文章的 excerpt 长度估算（excerpt * 5 作为粗略全文倍数），显示为 "X.X万"
- 连续写作天数：从今天往前回溯，连续有文章的天数

#### 2. 写作热力图（GitHub 风格）

- 展示最近 365 天的写作活动
- 用 CSS Grid 布局：53 列（周）× 7 行（天）
- 每个格子根据当天文章数量着色：0 篇灰色、1 篇浅绿、2 篇中绿、3+ 篇深绿
- 鼠标悬停显示日期和文章数（title 属性）
- 底部标注月份

#### 3. 月度趋势柱状图

- 最近 12 个月每月文章数量
- 纯 CSS 柱状图：每根柱子宽度均分，高度按比例
- 柱子上方显示数字，底部显示月份

#### 4. 标签分布

- 前 10 个最常用标签的横向条形图
- 每行：标签名 + 带颜色的 bar（宽度 = 该标签文章数 / 最大标签文章数 × 100%）+ 数量

**数据来源：** 所有数据来自文章列表加载后的 `articles` 数组（日期、标签、类型），不需要额外 API 调用。

**零第三方依赖：** 所有图表纯 CSS + HTML 实现。

### 受影响的文件

- 新建：`src/components/admin/AdminDashboard.tsx`
- 修改：`src/app/admin/page.tsx`（引入 Dashboard 组件，移除内联统计代码）

## 模块 D：Admin 路由密码保护

### 现状

任何人都能访问 `/admin/` 路由，只是实际操作需要 GitHub Token。页面内容（文章列表、标题等）对未授权访问者可见。

### 设计

在 admin layout 中添加密码门组件 `AdminAuth`：

**首次设置流程：**
1. 如果 localStorage 没有 `admin-password-hash`，在 settings 页面提示"设置管理密码"
2. 用户输入密码 → Web Crypto SHA-256 哈希 → 存入 `admin-password-hash`
3. 设置完成后当前 session 自动通过

**访问验证流程：**
1. 进入任何 `/admin/*` 页面
2. 检查 sessionStorage 是否有 `admin-authenticated` 标记
3. 如果没有：显示全屏密码输入框（覆盖页面内容）
4. 用户输入密码 → SHA-256 哈希 → 与存储的哈希比对
5. 匹配：sessionStorage 设置标记，显示页面内容
6. 不匹配：提示"密码错误"，可重试
7. 关闭浏览器 tab 后 sessionStorage 清空，下次需重新验证

**密码管理（settings 页面）：**
- 修改密码：输入旧密码验证后设置新密码
- 清除密码：输入当前密码后移除保护

**安全边界说明：** 这是前端密码门，可被浏览器开发者工具绕过。目的是防止无意访问，不是防御恶意攻击。真正的安全依赖 GitHub Token 的保护。

### 受影响的文件

- 新建：`src/components/admin/AdminAuth.tsx`（密码门 UI + 验证逻辑）
- 修改：`src/app/admin/layout.tsx`（用 AdminAuth 包裹子页面）
- 修改：`src/app/admin/settings/page.tsx`（添加密码设置/修改/清除 UI）

## 模块 E：健壮 frontmatter 解析

### 现状

admin 端（`admin/page.tsx` 和 `admin/edit/page.tsx`）各有一份自写的 `parseFrontMatter` 函数，只支持单行 `key: value` 和简单数组 `[a, b]`。不支持：
- 多行字符串值
- YAML block scalar（`|` 和 `>`）
- 嵌套对象
- 引号内含冒号的值

服务端 `content.ts` 使用 `gray-matter`（Node.js），浏览器端无法直接使用。

### 设计

引入 `js-yaml`（纯 JS，浏览器兼容）：

新建 `src/lib/frontmatter.ts`：

```typescript
import yaml from 'js-yaml';

interface ParsedFrontMatter {
  fm: Record<string, unknown>;
  body: string;
}

export function parseFrontMatter(content: string): ParsedFrontMatter {
  const match = content.match(/^---\n([\s\S]*?)\n---\n?([\s\S]*)$/);
  if (!match) return { fm: {}, body: content };
  
  try {
    const fm = yaml.load(match[1]) as Record<string, unknown>;
    return { fm: fm || {}, body: match[2] };
  } catch {
    return { fm: {}, body: content };
  }
}
```

- `admin/page.tsx` 和 `admin/edit/page.tsx` 中的 `parseFrontMatter` 函数删除，改为 `import { parseFrontMatter } from '@/lib/frontmatter'`
- 解析失败时 graceful fallback（返回空 fm + 原始内容），不抛出异常

### 受影响的文件

- 新建：`src/lib/frontmatter.ts`
- 修改：`src/app/admin/page.tsx`（删除本地 parseFrontMatter，改用 import）
- 修改：`src/app/admin/edit/page.tsx`（同上）
- 新增依赖：`js-yaml`、`@types/js-yaml`

## 已知限制

- **密码保护是前端方案**：可被开发者工具绕过，仅防无意访问
- **媒体库无引用检测**：删除图片时不检查是否有文章引用
- **总字数为估算值**：基于 excerpt 长度乘以倍数，非精确统计
- **热力图无文章详情**：悬停只显示日期和数量，不显示具体文章标题

## 不在 V3.2 范围内

- 编辑器预览模式（Tiptap 已所见即所得）
- 文章搬家（改类型/重命名）
- 标签管理（独立页面管理标签体系）
- 批量操作（多选 + 批量删除/移动）
- 图片引用检测和孤儿清理
