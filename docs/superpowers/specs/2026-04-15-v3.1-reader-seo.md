# V3.1 设计规格：读者体验 + SEO 基础

## 目标

提升博客读者的阅读体验和搜索引擎可见度。涵盖目录导航、前后篇导航、推荐阅读、阅读时间、标签可点击、RSS 订阅、OG 元标签、sitemap、favicon。

## 技术决策记录

| 决策 | 选项 | 选定 | 理由 |
|------|------|------|------|
| TOC 位置 | 右侧浮动 / 文章顶部 / 两者结合 | **右侧浮动** | 阅读时随时可见，不打断阅读流；移动端视口不够时隐藏 |
| TOC 实现 | 构建时提取 / 客户端 DOM 提取 | **客户端 DOM 提取** | HTML 已渲染好（`dangerouslySetInnerHTML`），从 DOM 中提取 h2/h3 最直接 |
| 当前章节高亮 | scroll 事件计算 / IntersectionObserver | **IntersectionObserver** | 性能好，不阻塞主线程 |
| 推荐算法 | 标签重叠度 / Fuse.js 标题搜索 / TF-IDF | **标签重叠度** | 零依赖，构建时确定结果，对 43 篇文章足够有效 |
| 标签点击行为 | 首页搜索 / 独立标签页 / URL 参数筛选 | **首页搜索** | 零成本复用 Fuse.js，不需要新页面新路由 |
| RSS 格式 | RSS 2.0 / Atom 1.0 / 两者 | **RSS 2.0** | 兼容性最广，实现最简单 |
| 阅读速度 | 400 字/分钟 / 300 字/分钟 | **400 字/分钟** | 中文阅读标准速率 |
| Favicon | 月亮 / 文字 / 灯笼 | **新月图形** | 取"月光集市"意象，简洁辨识度高，暗色主题呼应 |

## 模块 A：目录导航（TOC）

### 架构

新建 `TableOfContents` 客户端组件。文章页渲染后，从已挂载的 DOM 中提取 `h2`/`h3` 标签构建目录数据。使用 `IntersectionObserver` 监听各标题元素的可见性，高亮当前所在章节。

### 布局

文章内容区 `max-w-[980px]` 不变。右侧栏使用 `fixed` 定位在内容区右侧外面，通过 CSS 计算位置（`left: calc(50% + 490px + 40px)`），宽度 200px。当视口宽度不足时（< 1280px），通过媒体查询隐藏整个右侧栏。

右侧栏包含两部分：
1. **TOC 目录** — 限高 `max-height: 50vh`，超出部分滚动
2. **推荐阅读** — 紧跟 TOC 下方，展示 2-3 篇相关文章（详见模块 B）

这样推荐阅读在阅读过程中始终可见，不必等到文章读完。

### TOC 数据结构

```typescript
interface TocItem {
  id: string;      // heading 元素的 id（需确保 rehype 输出的标题有 id）
  text: string;    // 标题文本
  level: 2 | 3;    // h2 或 h3
}
```

### 标题 ID 生成

当前 markdown 管线（rehype-pretty-code + rehype-stringify）不自动给标题加 `id`。需要在 `markdown.ts` 的 unified 管线中添加 `rehype-slug` 插件，为每个 heading 自动生成 slug id。

### 依赖

```
rehype-slug  — 为标题元素自动添加 id 属性
```

### 受影响的文件

- 新建：`src/components/article/ArticleSidebar.tsx`（右侧栏容器：TOC + 推荐阅读）
- 新建：`src/components/article/TableOfContents.tsx`（TOC 组件）
- 修改：`src/lib/markdown.ts`（添加 rehype-slug 插件）
- 修改：`src/components/article/ArticleContent.tsx`（引入右侧栏，调整布局）

## 模块 B：前后篇导航 + 推荐阅读

### 前后篇导航

文章底部显示同类型内的上一篇和下一篇，按日期排序。上一篇 = 日期更早的相邻文章，下一篇 = 日期更晚的相邻文章。

实现方式：在各 `[slug]/page.tsx` 中调用 `getArticlesByType` 获取同类型文章列表（已按日期降序排列），找到当前文章索引，取前后。

### 推荐阅读

展示在右侧栏 TOC 下方（桌面端），2-3 篇相关文章。移动端不显示（随右侧栏一起隐藏）。

**推荐算法（标签重叠度）：**
1. 取当前文章的 tags
2. 遍历所有文章（跨类型），计算每篇与当前文章的共同标签数量
3. 排除自身
4. 按共同标签数降序，取 top 3
5. 相同重叠度时按日期降序（优先展示新文章）
6. 如果当前文章无标签或无任何重叠，fallback 到同类型最新 3 篇（排除自身）

**推荐计算时机：** 构建时在 `[slug]/page.tsx` 中计算。调用 `getAllArticles()` 获取全量文章列表，计算推荐结果，传给 `ArticleContent` 组件。

### 数据接口

```typescript
interface ArticleNavigation {
  prev: ArticleMeta | null;   // 同类型上一篇（日期更早）
  next: ArticleMeta | null;   // 同类型下一篇（日期更晚）
  related: ArticleMeta[];     // 推荐阅读（最多 3 篇）
}
```

### 展示

**右侧栏（桌面端）：**
```
┌─ 目录 ──────────┐
│  第一章          │
│  第二章          │ ← max-height: 50vh, 超出滚动
│  第三章          │
├─ 推荐阅读 ──────┤
│  文章标题1       │
│  文章标题2       │
│  文章标题3       │
└─────────────────┘
```

**文章底部（所有设备）：**
```
─────────────────────────
  ← 上一篇标题        下一篇标题 →
─────────────────────────
```

## 模块 C：阅读时间估算

### 计算方式

在 `content.ts` 的 `getArticle` 函数中计算阅读时间：

1. 从 markdown 原文中移除代码块、front matter 等非阅读内容
2. 统计中文字符（`/[\u4e00-\u9fff]/g`）和英文单词（`/[a-zA-Z]+/g`）
3. 阅读时间 = 中文字符数 / 400 + 英文单词数 / 200，向上取整，最少 1 分钟

### 类型扩展

在 `Article` 接口中添加 `readingTime: number`（分钟数）。`ArticleMeta` 不需要此字段（列表页不显示阅读时间）。

### 展示

在 `ArticleContent` 中，日期旁边显示"约 X 分钟"。

### 受影响的文件

- 修改：`src/lib/types.ts`（Article 接口添加 readingTime）
- 修改：`src/lib/content.ts`（getArticle 中计算阅读时间）
- 修改：`src/components/article/ArticleContent.tsx`（显示阅读时间）

## 模块 E：RSS 订阅

### 实现

新建构建脚本 `scripts/generate-rss.mjs`，遍历所有文章，输出 `public/feed.xml`。

RSS 2.0 格式，包含：
- `<channel>` 信息：标题、描述、链接、语言（zh-CN）、lastBuildDate
- 每篇文章一个 `<item>`：title、link、description（excerpt）、pubDate、guid、category（tags）
- 最多包含最近 50 篇文章（按日期降序）

在 `layout.tsx` 的 metadata 中添加 `alternates.types`，输出 `<link rel="alternate" type="application/rss+xml">` 标签。

build script 链式调用：`generate-search-index.mjs && generate-rss.mjs && next build`。

### 受影响的文件

- 新建：`scripts/generate-rss.mjs`
- 修改：`src/app/layout.tsx`（metadata 添加 RSS link）
- 修改：`package.json`（build script 加入 RSS 生成）
- 修改：`.gitignore`（添加 `public/feed.xml`）

## 模块 G：sitemap.xml + robots.txt

### 实现

新建 `scripts/generate-sitemap.mjs`，遍历所有文章生成 sitemap XML。包含：
- 首页 URL
- 所有文章 URL（`/{type}/{slug}/`）
- 每个 URL 包含 `<lastmod>`（文章日期）和 `<changefreq>`（monthly）

`public/robots.txt` 为静态文件：
```
User-agent: *
Allow: /
Sitemap: https://xiaoxiaoyi12.github.io/sitemap.xml
```

build script 链式调用加入 sitemap 生成。

## Build Script 最终形态

```json
"build": "node scripts/generate-search-index.mjs && node scripts/generate-rss.mjs && node scripts/generate-sitemap.mjs && next build"
```

四个步骤顺序执行：搜索索引 → RSS → sitemap → Next.js 构建。前三个脚本都输出到 `public/`，Next.js 构建时会将它们包含在静态输出中。

## 已知限制

- **TOC 仅桌面端可见**：视口宽度 < 1280px 时隐藏，移动端用户需手动滚动。这是有意取舍——移动端加 TOC 会压缩正文空间。如未来需要移动端 TOC，可增加顶部折叠方案。
- **推荐算法依赖标签质量**：如果文章标签为空或标签体系不一致，推荐效果会差。fallback 到同类型最新文章保底。
- **RSS 不含全文**：只包含 excerpt，避免 feed.xml 过大。如需全文 RSS，未来可加配置项。
- **无 og:image**：需要动态图片生成服务（如 Vercel OG 或 Satori），对静态导出不友好。V4 考虑。
