>
> 贯穿项目：**将 bookmark 阅读器升级为全栈应用**

## 学习哲学

在 AI 能写大部分代码的时代，不再按"Day 1 学语法、Day 2 学 API"的传统模式。而是：

1. **项目驱动**：围绕 bookmark 全栈化推进，遇到什么学什么
2. **原理优先**：不背 API 用法，理解"为什么这样做"
3. **AI 协作**：编码交给 AI，你负责设计、Review、决策、排查
4. **刻意练习**：每个阶段有"关掉 AI 自己写"的挑战，检验真实理解
5. **拆解先行**：每个任务先拆解再动手，刻意训练"把大问题变小"的能力

## 能力模型

```
你下指令 → AI 写代码 → 你 Review → 你做决策 → 你排查问题
   ①            ②          ③          ④           ⑤
```

重点培养 ①③④⑤，② 交给 AI。

## 问题拆解方法论

这是贯穿 90 天的核心训练。每次面对一个任务，用这个框架拆解：

### 万能拆解四步法

```
第 1 步：这个任务的最终产出是什么？（明确终点）
第 2 步：要达到终点，需要哪些"材料"？（找出依赖）
第 3 步：这些材料按什么顺序准备？（排依赖关系）
第 4 步：每个步骤的验证标准是什么？（怎么知道做对了）
```

### 示例：给 bookmark 加用户登录功能

```
❌ 错误拆解：1. 写登录页面 2. 写后端接口 3. 联调
  → 粒度太粗，每一步都不知道怎么动手

✅ 正确拆解：
  第 1 步 — 终点：用户能注册、登录、访问受保护页面
  第 2 步 — 材料：
    - 数据库：User 表（存什么字段？）
    - 后端：注册接口、登录接口、JWT 签发、验证中间件
    - 前端：注册页面、登录页面、token 存储、请求拦截器
  第 3 步 — 顺序：
    1. 设计 User 表结构 → 2. Prisma Schema + migrate
    → 3. 注册接口 → 4. 登录接口 + JWT
    → 5. 验证中间件 → 6. 前端页面 → 7. 联调
  第 4 步 — 验证标准：
    - 注册后数据库有记录？密码是哈希不是明文？
    - 登录返回 token？错误密码返回 401？
    - 带 token 能访问、不带 token 返回 403？
```

**每天的实战任务中，我会先给出"拆解提示"，你自己拆解，然后对比参考拆解。**
