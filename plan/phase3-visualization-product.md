### Day 47（周二）— SVG 原理 + D3 核心思想 🔨

#### 🧠 独立思考（30 分钟）

**先理解 SVG**（不写 D3 代码）：
- SVG 坐标系：原点在左上角，y 轴向下（和数学坐标系相反！）
- 基本图形：`<rect>`, `<circle>`, `<line>`, `<path>`, `<text>`
- 自己手写一个 SVG：画一个 200x100 的矩形 + 一个半径 30 的圆
- 问自己：如果要画一个柱状图，每根柱子是什么 SVG 元素？（rect）

**理解 D3 的核心思想**：
- D3 不是"画图表的库"，它是"数据驱动文档操作的库"
- 核心范式：**数据绑定到 DOM 元素**
  - Enter：数据多于元素 → 新增元素
  - Update：数据和元素对应 → 更新元素属性
  - Exit：元素多于数据 → 移除元素
- 这意味着：数据变了，图表自动变

#### 🤖 AI 辅助（30 分钟）

- 让 AI 写一个最简单的 D3 柱状图（5 根柱子，固定数据）
- 你逐行阅读代码，理解每一行在做什么

#### 👀 Review 要点

- SVG 坐标是否正确？（柱子是从底部向上长的，但 SVG y 轴向下——需要转换）
- D3 是否用了 Enter-Update-Exit 模式？
- 是否理解了 `d3.select().data().join()` 的含义？

### Day 48（周三）— D3 比例尺 + 坐标轴 🔨

#### 🧠 独立思考（25 分钟）

**理解比例尺的本质**：
- 比例尺 = **数据空间 → 像素空间的映射函数**
- 比如：阅读时长 0-120 分钟 → 像素 0-400px
- `scaleLinear`：连续数据（时长、数量）
- `scaleBand`：分类数据（星期一到星期日）
- `scaleTime`：时间数据（日期）

  - x 轴：日期 → `scaleTime` 或 `scaleBand`

- domain 是什么？range 是什么？

### Day 50（周五）— 原理日：可视化感知原理 📖

#### 🧠 独立思考（全天，60-90 分钟）

今天不写代码，学可视化设计原理。

**带着这些问题去学**：

1. **视觉编码通道**（20 分钟）：
   - 人类最精确感知的视觉通道排序：位置 > 长度 > 面积 > 颜色深浅 > 形状
   - 这就是为什么柱状图（长度编码）比饼图（面积+角度编码）更容易精确比较
   - 审查你 Day 48-49 做的图表，选择的视觉编码合理吗？

2. **颜色使用原则**（15 分钟）：
   - 不要用红绿表示好坏（色盲问题，影响 8% 男性）
   - 顺序数据用单色渐变（浅蓝到深蓝），分类数据用不同色相
   - 颜色不要超过 7 种（超过了人就分不清了）
   - 你的图表配色符合这些原则吗？

3. **常见可视化谎言**（15 分钟）：
   - 截断 Y 轴：让小差异看起来像大差异
   - 3D 饼图：视角导致面积失真
   - 面积误导：圆的半径翻倍，面积是 4 倍
   - 找 3 个网上的"糟糕图表案例"，分析为什么糟糕

4. **信息墨水比**（10 分钟）：
   - Edward Tufte 的核心思想：去掉图表中不传递信息的元素
   - 网格线需要吗？背景色需要吗？装饰性元素需要吗？
   - 审查你的图表，能删掉什么而不丢失信息？

5. **D3 vs ECharts：什么时候用谁？**（10 分钟）：
   - **ECharts**（配置式）：传入数据和配置对象 → 自动生成图表。适合标准图表（折线、柱状、饼图、热力图），开箱即用，上手快
   - **D3**（编程式）：手动操作 SVG 元素，完全控制每个像素。适合高度定制的图表（力导向图、自定义交互、非标准可视化）
   - 思考：bookmark 的仪表盘里，哪些图表用 ECharts 更高效？哪些必须用 D3？
   - 下周实战中会两个都用到，现在先建立选型意识

#### 🤖 AI 使用方式

- 你自己学完写完笔记后，让 AI 检查理解是否正确

#### 产出

- 一篇博客笔记《为什么你的图表不好看——可视化设计原则》
- 包含至少 3 个"糟糕图表案例"的分析

## 第 10 周（Day 60-65）：打磨 + D3/React 封装

### Day 60（周一）— D3 + React 封装模式 🔨

#### 🧠 独立思考（25 分钟）

**D3 和 React 有冲突**：
- D3 想操作 DOM（`d3.select().append()`）
- React 也想操作 DOM（虚拟 DOM diff）
- 两个都想控制 DOM，怎么办？

**两种模式**：
- **模式 A：D3 操作 DOM**：用 useRef 拿到 DOM 引用，D3 在 useEffect 中操作
  - 优点：能用 D3 全部功能
  - 缺点：绕过了 React 的声明式渲染
- **模式 B：React 渲染 SVG，D3 只做计算**：D3 只用来算比例尺、生成路径数据，React 渲染 SVG 元素
  - 优点：完全声明式，React 生态友好
  - 缺点：D3 的动画和交互不太好用

**自己想**：bookmark 的图表应该用哪种模式？为什么？

- 两种模式各试一个，你自己感受哪个更好维护

### Day 61（周二）— 图表动画与过渡 🔨

#### 🧠 独立思考（15 分钟）

**思考动画的目的**：
- 动画不是为了好看，是为了帮助用户理解数据变化
- 首次加载：柱子从 0 长到目标高度 → 用户注意到每根柱子的高度
- 数据更新：柱子平滑变化 → 用户能看出"哪个值变了"
- 过度动画反而分散注意力

**关键决策**：
- 动画时长多久合适？（200-500ms 比较舒服，超过 1s 就烦了）
- 什么图表需要动画？（折线图的路径绘制、柱状图的高度变化）
- 什么不需要？（Tooltip 的显示不需要动画延迟）

#### 🤖 AI 辅助（45 分钟）

- 让 AI 给柱状图添加首次加载的入场动画（从底部升起）
- 让 AI 给折线图添加路径绘制动画
- 让 AI 实现切换时间范围时的平滑过渡

#### 👀 Review 清单

- [ ] 动画是否流畅？（60fps？有没有卡顿？）
- [ ] 动画时长是否合理？
- [ ] 用户快速切换时，动画是否会叠加出 bug？
- [ ] 动画是否可以被系统的"减少动画"偏好禁用？

### Day 63（周四）— 性能优化 🔨

#### 🧠 独立思考（20 分钟）

**思考图表性能问题**：
- 如果有 365 天的数据（一年热力图），渲染 365 个 SVG 元素会卡吗？
- 如果力导向图有 200 个节点 + 500 条边，动画会卡吗？
- React 重渲染时，图表是否会不必要地重新绑定数据？

**优化策略**：
- `React.memo`：图表组件的 props 没变就不重渲染
- `useMemo`：缓存计算结果（比例尺、路径数据）
- Canvas 替代 SVG：数据点超过 1000 时考虑
- 数据采样：展示 365 天数据时，是否可以按周聚合？

#### 🤖 AI 辅助（40 分钟）

- 让 AI 对图表组件添加 React.memo + useMemo 优化
- 让 AI 用 Lighthouse 跑分仪表盘页面
- 如果有必要，让 AI 实现数据采样逻辑

#### 👀 Review 清单

- [ ] React.memo 的比较函数是否正确？（浅比较够吗？）
- [ ] useMemo 的依赖数组是否正确？
